Repository:
loofiboss-bit/plasma-ai-usage-monitor

Task:
Design and implement a UI redesign for v6.0.0 of the KDE Plasma 6 widget, with focus on a cleaner, more premium, more operator-friendly Kirigami/Plasma experience.

Important product constraints:
- Keep this a native Plasma/Kirigami/QML widget.
- Do NOT redesign it into a web dashboard or SaaS-style control panel.
- Do NOT introduce a new backend architecture, PostgreSQL, login system, multi-user state, or remote team service.
- Preserve all existing backend contracts, provider/tool wiring, data models, polling behavior, and current feature coverage unless a UI change absolutely requires a small safe adaptation.
- Favor QML/UI refactoring over C++ changes.
- Maintain accessibility, keyboard navigation, theming, i18n friendliness, and narrow popup responsiveness.

Current UI context to respect:
- The widget already has Live / History / Analyst surfaces.
- It already uses modular UI components such as FullRepresentation.qml, ProviderCard.qml, SubscriptionToolCard.qml, CostSummaryCard.qml, UsageChart.qml, MultiSeriesChart.qml, TrendSummary.qml, and config*.qml files.
- Recent versions already added “Needs Attention”, collapsible cards, and composition-root cleanup, so this redesign should go beyond polish and establish a stronger visual system and information architecture.

Redesign goals for v6.0.0:
1. Create a cohesive visual language for the entire plasmoid:
   - consistent card grammar
   - calmer spacing
   - clearer section hierarchy
   - better use of Kirigami.Theme roles
   - more polished icon/text balance
   - less visual noise
   - stronger “premium KDE widget” feel

2. Redesign Live view:
   - Make it feel like a true dashboard, not a long stack of unrelated blocks.
   - Reorganize into a clearer order:
     a) top summary / health strip
     b) actionable “Needs Attention” area
     c) high-value rollup cards
     d) providers
     e) subscription tools
   - Reduce duplication between summary chips and card headers.
   - Improve scanability when many providers/tools are enabled.
   - Make healthy states quieter and warning/error states more visually intentional.

3. Redesign ProviderCard.qml:
   - Keep collapse/expand behavior, but improve the visual rhythm.
   - Use better grouping for:
     - header identity
     - connection status
     - usage/cost metrics
     - budgets
     - rate limits
     - last refresh
   - Make cost and budget information easier to read at a glance.
   - Make error states feel actionable, not cluttered.
   - Reduce cramped label/value layouts on narrow widths.

4. Redesign SubscriptionToolCard.qml:
   - Align its visual system with ProviderCard, but keep a distinct tool-monitor identity.
   - Improve hierarchy for plan, status, usage bars, reset timing, credits, sync state, and warnings.
   - Make action buttons feel integrated and less tacked-on.
   - Reduce card height inflation where possible.

5. Redesign History and Analyst views:
   - Improve control layout and readability.
   - Reduce the “toolbar dump” feel in history controls.
   - Make charts and ranking panels look more intentional and less dev-tool-ish.
   - Improve empty/loading/no-data states.
   - Ensure compare mode feels like a first-class view, not an add-on.

6. Redesign onboarding / zero-state / disconnected states:
   - Make first-run setup feel more polished and less boxy.
   - Improve empty states so they clearly guide the user without feeling generic.
   - Keep setup practical and local-first.

7. Redesign config screens if needed:
   - Review configGeneral.qml, configProviders.qml, configAlerts.qml, configBudget.qml, configSubscriptions.qml, and configHistory.qml.
   - Bring them closer to the new v6 visual language.
   - Improve field grouping, provider-specific sections, and visual clarity without overcomplicating the KCM.

8. Responsive behavior:
   - Improve narrow popup behavior across Live, History, Analyst, and cards.
   - Remove awkward wrapping, cramped metrics, and overly long horizontal control rows where possible.
   - Prefer adaptive layouts over brute-force shrinking.

9. KDE/Plasma quality:
   - Use Kirigami and Plasma components idiomatically.
   - Respect theme contrast in both light and dark modes.
   - Avoid hardcoded visual hacks when a theme-aware approach exists.
   - Preserve or improve accessibility names, focus order, and keyboard navigation.

Implementation expectations:
- Start by auditing the current UI files and propose a concise redesign plan.
- Then implement the redesign directly in the existing QML structure.
- Split oversized QML into additional reusable components if that improves maintainability.
- Keep the architecture clean; do not re-monolith main.qml.
- Prefer reusable helper components where repeated visual patterns exist.
- Update screenshots/docs/changelog/version notes for the redesign.
- Add or adjust any tests or smoke checks that are appropriate for UI-safe validation.
